Monter un cluster Kubernetes, ce n’est pas juste lancer un kubectl create cluster. C’est assembler des dizaines de briques — réseau, stockage, sécurité, monitoring, déploiement — pour obtenir une plateforme fiable et opérable au quotidien. Voici le retour d’expérience de la construction from scratch d’un cluster de production sur OVHcloud Managed Kubernetes (MKS).

Le contexte #

L’objectif : fournir une plateforme mutualisée capable d’héberger plusieurs projets applicatifs pour des équipes de développement, avec des environnements de dev, recette et production sur le même cluster. Le tout en souveraineté européenne — d’où le choix d’OVHcloud plutôt qu’un hyperscaler américain.

Les contraintes dès le départ :

  • Infrastructure as Code — tout doit être versionné, reproductible, auditable.
  • Sécurité renforcée — chiffrement, WAF, audit, scan de vulnérabilités.
  • Observabilité complète — métriques, logs, alerting, monitoring externe.
  • Autonomie des équipes — les développeurs déploient sans intervention infra.
  • Maîtrise des coûts — extinction automatique des environnements hors heures ouvrées.

Provisionnement avec OpenTofu #

Le cluster et son réseau sont provisionnés via OpenTofu (fork open source de Terraform). Trois modules structurent l’infra :

  • network — réseau privé VRack avec un sous-réseau dédié au cluster.
  • gateway — passerelle managée pour la sortie internet des pods.
  • kubernetes — cluster MKS avec un node pool de machines b3-16.

Le tout est découpé par environnement (production, hors-prod, outillage) avec des variables dédiées par contexte. Le state est stocké de manière sécurisée et les imports de ressources existantes sont documentés pour les reprises en main.

L’architecture applicative du cluster #

Une fois le cluster debout, il faut l’habiller. Voici les briques déployées, toutes gérées via Helm et versionnées en Git.

Ingress et TLS #

Traefik en tant qu’ingress controller, déployé en haute disponibilité avec deux réplicas et une anti-affinité de pods. Les certificats TLS sont gérés par cert-manager avec validation DNS via l’API OVH, ce qui permet les certificats wildcard Let’s Encrypt sans exposer de ports supplémentaires.

Un contrôleur Reflector propage automatiquement les secrets TLS dans les namespaces applicatifs — les équipes n’ont pas à se soucier de la gestion des certificats.

Sécurité — défense en profondeur #

La sécurité du cluster repose sur plusieurs couches complémentaires :

  • CrowdSec — WAF intégré à Traefik via le bouncer plugin. L’AppSec applique les règles OWASP CRS en virtual patching. Les IP malveillantes sont bloquées en amont, alimentées par la threat intelligence communautaire de CrowdSec.
  • Kyverno — moteur de policies Kubernetes en mode audit. Interdit les workloads dans le namespace default, nettoie automatiquement les pods orphelins et les ReplicaSets inutilisés.
  • Trivy Operator — scan continu des images déployées sur le cluster. Les vulnérabilités critiques déclenchent des alertes par email, routées vers l’équipe responsable du namespace concerné.
  • Falco — audit des appels API Kubernetes via le plugin OVH. Détecte les accès suspects aux secrets, les modifications de RBAC non autorisées, les exécutions de shell dans les pods.
  • Sealed Secrets — tous les secrets sont chiffrés dans Git. Seul le contrôleur Sealed Secrets dans le cluster peut les déchiffrer.

Observabilité #

La stack monitoring s’articule autour de trois piliers :

  • Métriques — Prometheus scrape les composants du cluster et les applications. Grafana fournit les dashboards avec SSO via Keycloak. AlertManager route les alertes par email avec des templates personnalisées et un routage par namespace.
  • Logs — Promtail collecte les logs de tous les pods avec un pipeline de parsing JSON structuré. Loki les stocke et les rend requêtables depuis Grafana.
  • Monitoring externe — Blackbox Exporter surveille la disponibilité des services exposés (HTTPS) avec des alertes sur les temps de réponse et l’expiration des certificats SSL.

GitOps avec ArgoCD #

Les déploiements applicatifs passent par ArgoCD : les équipes poussent leur code, la CI construit l’image, et ArgoCD Image Updater détecte la nouvelle version et met à jour le déploiement automatiquement. Quatre niveaux de RBAC (admin, développeur, chef de projet, lecture seule) sont synchronisés via Keycloak.

Les développeurs ont la main sur le sync et les overrides de leurs applications, mais pas sur la création ou suppression de projets ArgoCD — c’est l’équipe infra qui garde le contrôle du périmètre.

Stockage multi-tiers #

OVHcloud propose plusieurs types de stockage, tous exploités ici :

Type Service OVH Usage
Block (RWO) Cinder High Speed / Classic Prometheus, Grafana, AlertManager
Object (S3) Object Storage S3 Backups Velero, stockage applicatif
Fichier (RWX) NAS-HA via CSI NFS Loki, données partagées entre pods

Les StorageClasses sont configurées en Retain pour éviter la suppression accidentelle des volumes persistants.

Opérations #

  • Velero — sauvegarde quotidienne complète du cluster vers le stockage objet OVH, avec backup filesystem (Kopia) pour les volumes NFS. Rétention de 72 heures.
  • kube-green — suspend automatiquement les environnements de développement et recette en dehors des heures ouvrées (22h → 6h, lundi-vendredi). Sur un cluster mutualisé, l’économie de ressources est significative.
  • External Secrets Operator — synchronise les secrets depuis les variables CI/CD GitLab vers le cluster, évitant la duplication manuelle.

Les choix structurants #

Pourquoi OVHcloud MKS ? #

  • Souveraineté — données hébergées en France (Strasbourg), conformité RGPD native.
  • Kubernetes managé — le control plane est opéré par OVH, ce qui réduit la charge d’exploitation.
  • VRack — réseau privé L2 performant entre les nœuds, sans surcoût.
  • Coût maîtrisé — le control plane MKS est gratuit, on ne paie que les nœuds et le stockage.

Les limites rencontrées #

Quelques spécificités du MKS OVH qui ont demandé des adaptations :

  • Pas d’accès au control plane — impossible de scraper les métriques de l’API server, du scheduler ou de l’etcd directement. Le monitoring se concentre sur le kubelet et les composants applicatifs.
  • Falco en mode dégradé — pas d’accès au kernel des nœuds managés. Falco fonctionne en Deployment (pas DaemonSet) via le plugin d’audit OVH plutôt qu’en interception syscall.
  • Pas de LoadBalancer natif intégré — Traefik gère tout le trafic entrant via NodePort + Proxy Protocol.

Pourquoi tout en Helm plutôt qu’en manifestes bruts ? #

Avec plus de vingt composants à gérer, Helm apporte :

  • Versioning — chaque composant a une version pinée, les mises à jour sont tracées en Git.
  • Templating — les values files sont le seul point de configuration, tout le reste est géré par les charts upstream.
  • Reproductibilité — un helm upgrade sur un nouveau cluster reconstruit la même stack.

Ce que ça donne au quotidien #

Une fois la plateforme en place, le workflow pour une nouvelle application est simple :

  1. L’équipe dev crée son projet GitLab avec un Dockerfile et un chart Helm.
  2. La CI GitLab construit et pousse l’image sur le registre.
  3. L’infra crée le namespace et le projet ArgoCD correspondant.
  4. ArgoCD détecte le chart, déploie l’application, et met à jour automatiquement les versions d’images.
  5. Les métriques et logs sont immédiatement disponibles dans Grafana.
  6. Les alertes de sécurité (Trivy, CrowdSec, Falco) sont routées vers les bonnes équipes.
  7. Les sauvegardes tournent, les environnements s’éteignent la nuit, les certificats se renouvellent.

L’équipe infra intervient pour les opérations transverses (mises à jour du cluster, ajout de composants, gestion des politiques de sécurité), mais les équipes applicatives sont autonomes sur leur périmètre.

Conclusion #

Monter un cluster Kubernetes de production, ce n’est pas le cluster lui-même qui est complexe — c’est tout ce qu’on met autour. Le réseau, la sécurité, l’observabilité, le GitOps, les sauvegardes, l’optimisation des coûts : chaque brique prise isolément est simple, mais c’est leur intégration cohérente qui fait la différence entre un cluster “qui tourne” et une plateforme réellement opérable.

Le choix d’OVHcloud s’est avéré pertinent pour le contexte : souveraineté, coût maîtrisé, et un Kubernetes managé qui fait le travail. Les quelques limitations (accès au control plane, Falco sans kernel) se contournent sans compromis majeur sur la sécurité ou l’observabilité.

L’investissement initial dans l’IaC et l’automatisation se rentabilise dès le deuxième projet embarqué sur la plateforme.
